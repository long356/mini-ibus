## 一、项目目标和范围

### 1.1 你要做出来的东西

3 台树莓派 + 各种链路连接：

```
pi-A ─── WiFi ─── pi-B ─── USB ─── pi-C
```

- 各自跑一个 mini-ibusd 守护进程
- 各自跑几个业务进程
- 业务进程能跨设备调 RPC、发 pub/sub
- 拔掉 USB 后自动绕路（如果有 WiFi 备用路径）
- 插上后自动恢复

### 1.2 保留的亮点

| 亮点 | 实现方式 |
| --- | --- |
| NodeAddress = (role, name) | uint32 编码 |
| Pub/Sub + RPC 双模型 | 简单 IDL |
| MsgDispatcher 多 lane 分发 | 4 个线程池 |
| 中心化 ibusd + 路由表 | 每设备一个 ibusd |
| **多设备拓扑合并** | ibusd 之间 gossip + Dijkstra |
| 心跳 + 断链自动重路由 | 1秒心跳 + 5秒超时 |

### 1.3 砍掉的东西

- ❌ SPI/UART 等嵌入式链路（只支持 UDS/TCP）
- ❌ ReliableTunnel（TCP 已经可靠）
- ❌ 消息切片（限制 64KB）
- ❌ 多产品配置（只支持单一拓扑配置文件）
- ❌ 同步等待（RPC 一律异步 + future）

- C++17 (GCC 9+ 或 Clang 10+)

- 标准库 thread/mutex/condition_variable

## 四、协议设计

### 4.1 消息格式

每条消息分成两部分：**Link Header**（固定 16 字节）+ **Payload**（Protobuf）。

```cpp
// message_codec.h
struct LinkHeader {
  uint8_t magic[2];      // "MI" 防止解包错位
  uint8_t version;        // 协议版本
  uint8_t type;           // 消息类型（见下）
  uint32_t src_addr;      // 源 NodeAddress
  uint32_t dst_addr;      // 目标 NodeAddress
  uint32_t payload_size;
} __attribute__((packed));
static_assert(sizeof(LinkHeader) == 16);

enum class MessageType : uint8_t {
  REGISTER = 1,     // 业务进程注册（业务 → ibusd）
  RPC_REQUEST = 2,  // RPC 请求
  RPC_RESPONSE = 3, // RPC 响应
  PUBLISH = 4,      // pub/sub 消息
  SUBSCRIBE = 5,    // 订阅注册
  HEARTBEAT = 6,    // 心跳
  TOPOLOGY = 7,     // ibusd 之间的拓扑广播
};
```



## 五、NodeAddress 编码

```cpp
// core/address.h
#pragma once
#include <cstdint>
#include <string>

namespace minibus {

enum class Role : uint16_t {
  NONE = 0,
  CAMERA = 1,
  SCREEN = 2,
  BOX = 3,
  GUI = 4,
};

enum class Name : uint16_t {
  NONE = 0,
  MEDIAD = 1,
  HALD = 2,
  INS_GUI = 3,
  CAMERA_SVC = 4,
};

class Address {
public:
  static uint32_t Pack(Role r, Name n) {
    return (uint32_t(r) << 16) | uint32_t(n);
  }
  static Role GetRole(uint32_t a) { return Role(a >> 16); }
  static Name GetName(uint32_t a) { return Name(a & 0xFFFF); }
  static std::string ToString(uint32_t a);
};

} // namespace minibus
```

## 六、ibusd 内部架构

这是整个系统的大脑。

### 线程模型

```
┌─────────────────── 一个 ibusd 进程 ──────────────────┐
│                                                       │
│  ┌── 本地业务连接 ──┐    ┌── peer ibusd 连接 ──┐      │
│  │ UDS Listener     │    │ TCP Listener         │      │
│  │ /tmp/mibus.sock  │    │ 0.0.0.0:7654         │      │
│  └────────┬─────────┘    └──────────┬───────────┘      │
│           │                         │                  │
│           ▼                         ▼                  │
│  ┌─────────────────────────────────────────────────┐   │
│  │         IO 线程（epoll loop）                    │   │
│  │         - 收消息 → 入 message_queue              │   │
│  └────────────────────────┬────────────────────────┘   │
│                           │                            │
│                           ▼                            │
│  ┌─────────────────────────────────────────────────┐   │
│  │         Dispatcher 线程                          │   │
│  │         - 从 queue 拿消息                        │   │
│  │         - 按 dst_addr 查 route_table             │   │
│  │         - 路由到本地连接 / peer 连接              │   │
│  └─────────────────────────────────────────────────┘   │
│                                                       │
│  ┌─────────────────────────────────────────────────┐   │
│  │         心跳 + 拓扑线程（定时器）                 │   │
│  │         - 1秒：发心跳                            │   │
│  │         - 每5秒：广播本地拓扑                     │   │
│  │         - 拓扑变化时：触发 merge                  │   │
│  └─────────────────────────────────────────────────┘   │
│                                                       │
│  ┌─────────────────────────────────────────────────┐   │
│  │         Topo Merge 线程（条件变量唤醒）           │   │
│  │         - 合并本地+所有外部拓扑                   │   │
│  │         - 跑 Dijkstra                            │   │
│  │         - 更新共享内存路由表                      │   │
│  └─────────────────────────────────────────────────┘   │
│                                                       │
└───────────────────────────────────────────────────────┘
```

**总共 4-5 个线程**，互相通过队列 + 条件变量通信。

### 6.1 关键数据结构

```cpp
// ibusd/topology.h
namespace minibus::ibusd {

// 一条 link 边
struct LinkEdge {
  Role peer_role;
  uint32_t medium;      // 0=TCP, 1=USB, 2=WIFI
  uint32_t weight;
  uint64_t last_active_ms;
};

// 一个 role 节点
struct TopoNode {
  Role role;
  std::vector<LinkEdge> edges;
};

// 本地拓扑（这台设备能直连的）
struct LocalTopology {
  Role self_role;
  uint64_t version{0};
  std::map<Role, LinkEdge> peer_links; // role → link 信息
};

// 全局拓扑（合并后的）
class GlobalTopology {
public:
  void AddNode(Role r);
  void AddEdge(Role from, Role to, uint32_t weight, uint32_t medium);

  // 跑 Dijkstra，返回 from 到所有节点的下一跳
  std::map<Role, NextHop> CalculateShortestPaths(Role from);

private:
  std::map<Role, std::set<Role>> adjacency_;
  std::map<std::pair<Role, Role>, uint32_t> edge_weights_;
};

struct NextHop {
  Role next_hop;          // 下一跳的 role
  uint32_t total_weight;
  uint32_t hop_count;
};

} // namespace minibus::ibusd
```

### 6.2 路由表（共享内存）

这是 IBus 的精髓——业务进程不需要走 RPC 问 ibusd，而是**直接读共享内存的路由表**。

```cpp
// core/route_table.h
namespace minibus {

// 共享内存中的路由表条目
struct RouteEntry {
  uint32_t dst_addr;     // 目标完整地址
  uint32_t next_hop;     // 下一跳（如果是本机就是 dst 自己）
  uint32_t medium;       // 走哪条链路
  bool reachable;
};

// 共享内存中的整张表（fixed size）
struct RouteTableShm {
  std::atomic<uint64_t> version;     // 每次更新自增
  std::atomic<uint32_t> entry_count;
  RouteEntry entries[1024];
};

class RouteTableReader {
public:
  bool Init(const std::string& shm_name);
  std::optional<RouteEntry> Lookup(uint32_t dst_addr) const;
private:
  RouteTableShm* shm_;
};

class RouteTableWriter {
public:
  bool Init(const std::string& shm_name);
  void Update(const std::vector<RouteEntry>& entries);
private:
  RouteTableShm* shm_;
};

} // namespace minibus
```

**为什么用共享内存？**
- 业务进程频繁查路由（每发一条消息都查），不能走 IPC
- 路由表本身变化不频繁（秒级），适合"一处写多处读"
- 用 `version` + `atomic` 实现简单的无锁读

读端代码（业务进程）：
```cpp
auto entry = route_table_reader.Lookup(target_addr);
if (entry && entry->reachable) {
  // 把消息发给 ibusd（UDS），ibusd 再按 next_hop 转发
  SendToLocalIbusd(msg);
}
```

## 七、多设备拓扑合并算法（核心亮点）

### 7.1 整体流程

```
设备 A 的 ibusd                       设备 B 的 ibusd
─────────────────                     ─────────────────
本地链路状态变化
（USB 插上 / WiFi 断开）
    ↓
UpdateLocalTopology()
    ↓
version++
    ↓
BroadcastLocalTopology() ──────────→ 收到 LocalTopology
                                      ↓
                                   StoreExternalTopology(A)
                                      ↓
                                   NotifyMergeThread()
                                      ↓
                                   MergeAllTopologies()
                                      ↓
                                   CalculateShortestPaths(self=B)
                                      ↓
                                   UpdateRouteTable()

反方向：B 自己的变化也通过同样路径传给 A
```

### 7.2 关键算法

**步骤 1：收集本地拓扑**
```cpp
// ibusd/topology.cc
void IbusdServer::UpdateLocalTopology() {
  LocalTopology new_local;
  new_local.self_role = self_role_;
  new_local.version = ++local_version_;

  for (auto& [peer_role, peer_conn] : peer_connections_) {
    if (peer_conn->IsAlive()) {
      LinkEdge edge;
      edge.peer_role = peer_role;
      edge.medium = peer_conn->GetMedium();
      edge.weight = ComputeWeight(peer_conn);
      edge.last_active_ms = NowMs();
      new_local.peer_links[peer_role] = edge;
    }
  }

  {
    std::lock_guard<std::mutex> lock(topo_mutex_);
    local_topology_ = std::move(new_local);
  }

  NotifyMergeThread();           // 唤醒 merge 线程
  BroadcastLocalTopology();      // 通知所有 peer
}
```

**步骤 2：广播本地拓扑**
```cpp
void IbusdServer::BroadcastLocalTopology() {
  proto::LocalTopology proto_topo = SerializeLocalTopology(local_topology_);

  LinkHeader header;
  header.type = uint8_t(MessageType::TOPOLOGY);
  header.src_addr = self_ibusd_addr_;
  header.dst_addr = 0; // 广播

  std::string payload = proto_topo.SerializeAsString();
  header.payload_size = payload.size();

  for (auto& [peer_role, peer_conn] : peer_connections_) {
    peer_conn->Send(header, payload);
  }
}
```

**步骤 3：接收外部拓扑**
```cpp
void IbusdServer::OnPeerTopologyReceived(Role peer_role,
    const proto::LocalTopology& topo) {
  {
    std::lock_guard<std::mutex> lock(topo_mutex_);
    // 版本号比较：旧的丢弃
    auto& stored = external_topologies_[peer_role];
    if (topo.version() <= stored.version) {
      return;
    }
    stored = ParseLocalTopology(topo);
  }
  NotifyMergeThread();
}
```

**步骤 4：合并 + 计算路由（核心）**
```cpp
void IbusdServer::MergeAndComputeRoutes() {
  // 1. 拷贝当前所有拓扑数据（短时持锁）
  LocalTopology local_snap;
  std::map<Role, LocalTopology> external_snap;
  {
    std::lock_guard<std::mutex> lock(topo_mutex_);
    local_snap = local_topology_;
    external_snap = external_topologies_;
  }

  // 2. 构建全局图（合并所有节点的本地视图）
  GlobalTopology global;
  // 加入本地拓扑的边
  for (auto& [peer_role, edge] : local_snap.peer_links) {
    global.AddEdge(self_role_, peer_role, edge.weight, edge.medium);
  }
  // 加入所有外部拓扑的边
  for (auto& [publisher_role, topo] : external_snap) {
    for (auto& [peer_role, edge] : topo.peer_links) {
      global.AddEdge(publisher_role, peer_role, edge.weight, edge.medium);
    }
  }

  // 3. 跑 Dijkstra：算出从 self 到每个 role 的最短路径
  auto next_hops = global.CalculateShortestPaths(self_role_);

  // 4. 构建路由表条目
  std::vector<RouteEntry> entries;
  for (auto& [target_role, hop_info] : next_hops) {
    for (auto& target_name : known_names_in_role[target_role]) {
      RouteEntry entry;
      entry.dst_addr = Address::Pack(target_role, target_name);
      entry.next_hop = uint32_t(hop_info.next_hop);
      entry.reachable = true;
      entries.push_back(entry);
    }
  }

  // 5. 写入共享内存
  route_table_writer_.Update(entries);
}
```

### 7.3 Dijkstra 实现

```cpp
// ibusd/route_calculator.cc
std::map<Role, NextHop> GlobalTopology::CalculateShortestPaths(Role from) {
  std::map<Role, uint32_t> dist;
  std::map<Role, Role> prev;

  // 优先队列：(距离, 节点)
  using PQItem = std::pair<uint32_t, Role>;
  std::priority_queue<PQItem, std::vector<PQItem>, std::greater<>> pq;

  dist[from] = 0;
  pq.push({0, from});

  while (!pq.empty()) {
    auto [d, u] = pq.top(); pq.pop();
    if (d > dist[u]) continue;

    for (Role v : adjacency_[u]) {
      uint32_t w = edge_weights_[{u, v}];
      uint32_t nd = d + w;
      if (!dist.count(v) || nd < dist[v]) {
        dist[v] = nd;
        prev[v] = u;
        pq.push({nd, v});
      }
    }
  }

  // 回溯：对每个目标，找"从 from 出发的第一跳"
  std::map<Role, NextHop> result;
  for (auto& [target, d] : dist) {
    if (target == from) continue;
    Role hop = target;
    while (prev[hop] != from) {
      hop = prev[hop];
    }
    result[target] = {hop, d, /*hop_count=*/...};
  }
  return result;
}
```

### 7.4 用具体例子验证

**拓扑：** A ── B ── C，A 和 C 之间没有直连。

**初始 ibusd-A 看到的 local_topology：**
```
{ self: A, peer_links: { B: {weight: 1, medium: WIFI} } }
```

**ibusd-A 广播给 B。**

**最终所有人收到的拓扑信息：**
```
local_topology(A): { A→B }
external_topology(B): { B→A, B→C }  ← B 看到的
external_topology(C): { C→B }
```

**合并后的全局图：**
```
A ─weight=1─ B ─weight=1─ C
```

**ibusd-A 跑 Dijkstra：**
```
A→A: dist=0
A→B: dist=1, next_hop=B
A→C: dist=2, next_hop=B  ← 关键！去 C 要先走 B
```

**ibusd-A 的路由表：**
```
dst=GUI@C → next_hop=B, medium=WIFI
dst=HALD@C → next_hop=B, medium=WIFI
dst=HALD@B → next_hop=B, medium=WIFI
```

**当业务发消息：**
```cpp
client.Call(Address::Pack(GUI, C), method, args);
//  ↓ 查路由表：next_hop=B
//  ↓ 发给 ibusd-A
//  ↓ ibusd-A 看路由表，发给 ibusd-B（通过 WIFI 链路）
//  ↓ ibusd-B 收到，看到 dst=GUI@C，查自己的路由表 next_hop=C
//  ↓ ibusd-B 转发给 ibusd-C
//  ↓ ibusd-C 收到，dst 是本地，转给业务进程
```

完美的 hop-by-hop 路由。

### 7.5 故障处理

**场景：A-B 之间的 WIFI 断了**

1. ibusd-A 的心跳线程发现 B 没回心跳，标记 link 为 dead
2. ibusd-A 更新 local_topology（删除 A→B 边），version++
3. ibusd-A 广播新的 local_topology 到所有还能连上的 peer
4. 如果 A 和 C 之间还有备用链路（比如蓝牙），那 A 通过 C 把新拓扑传给 B
5. 所有 ibusd 重新跑 Dijkstra
6. 业务进程下次发消息时读到新路由表，自动绕路

**关键设计：** local_topology 用 **version 号** 防止旧消息覆盖新的。

```cpp
  client.Run();
}
```

### 8.2 MsgDispatcher 在客户端

业务进程内部用 MsgDispatcher 把 ibusd 推过来的消息分到 4 个 lane：

```cpp
class Client {
private:
  MsgDispatcher dispatcher_; // 4 个 looper

  void OnMessageFromIbusd(Message msg) {
    switch (msg.type) {
    case PUBLISH:
      if (IsUrgentTopic(msg.topic)) {
        dispatcher_.PostUrgent([=]{ topic_handlers_[topic](msg); });
      } else if (IsBlockingTopic(msg.topic)) {
        dispatcher_.PostBlocking([=]{ topic_handlers_[topic](msg); });
      } else {
        dispatcher_.Post([=]{ topic_handlers_[topic](msg); });
      }
      break;
    case RPC_REQUEST:
      dispatcher_.PostBlocking([=]{
        auto result = rpc_handlers_[svc][method](msg.args);
        SendResponse(msg.sequence, result);
      });
      break;
    }
  }
};
```

## 九、关键工程细节

### 9.1 ibusd 之间的连接建立

**问题**：谁主动连谁？

**方案**：用**配置文件**显式声明：

```yaml
# /etc/mini-ibus/config.yaml
self_role: CAMERA
listen_addr: 0.0.0.0:7654
listen_uds: /tmp/mini-ibus.sock
peers:
  - role: SCREEN
    address: 192.168.1.10:7654
    medium: WIFI
    weight: 1
  - role: BOX
    address: 192.168.1.11:7654
    medium: WIFI
    weight: 2
```

启动时按配置主动连。**避免动态发现的复杂性**。

### 9.2 业务节点的可达性

ibusd 只知道 role 级别的路由，但消息的 dst 是完整 `(role, name)`。怎么办？

**方案**：业务节点 Register 时上报自己的完整地址，ibusd 周期性把"我这台设备上有哪些业务节点"也广播出去。

广播内容：
```
LocalTopology { ... 链路边 ... }
NodeRegistrations: [
  { addr: (CAMERA, MEDIAD), services: [...], topics: [...] },
  { addr: (CAMERA, HALD), services: [...], topics: [...] },
]
```

所有 ibusd 都维护一个 `address → role` 的映射，转发时先看目标 role 然后查 next_hop。

### 9.3 心跳协议

```cpp
// ibusd 之间每秒发一次心跳
void HeartbeatThread::Run() {
  while (running_) {
    for (auto& [peer_role, conn] : peer_connections_) {
      HeartbeatPayload hb;
      hb.timestamp_us = NowUs();
      hb.seq = ++hb_seq_;
      conn->SendHeartbeat(hb);
    }

    // 检查超时：5 秒没收到心跳就标记 dead
    auto now = NowMs();
    for (auto& [peer_role, conn] : peer_connections_) {
      if (now - conn->LastRecvMs() > 5000) {
        conn->MarkDead();
        NotifyLocalTopologyChanged();
      }
    }

    std::this_thread::sleep_for(std::chrono::seconds(1));
  }
}
```

### 9.4 拓扑传播的环路问题

**场景**：A 广播给 B，B 又转发给 A，会不会循环？

**解决**：每个 LocalTopology 有 `publisher_role` 字段，ibusd 不存储自己 publish 的，也不转发别人 publish 的（只用于 merge）。第一版**先不做转发**，所有 ibusd 之间假设直连，简化设计。

### 9.5 启动顺序

1. ibusd 启动
2. 创建共享内存路由表（初始为空）
3. 启动 UDS Listener（等业务连）
4. 主动连接配置文件里的 peer ibusd
5. 等 peer 连上后，交换拓扑
6. 业务进程启动，连本地 ibusd
7. 业务进程注册自己
8. ibusd 把节点信息加进 LocalTopology，再次广播

## 十三、踩坑提示

### 13.1 启动顺序

A 启动时 B 还没起，A 连不上 B 会怎样？

**方案**：peer 连接用**自动重连**，每 3 秒重试一次。不要让 ibusd 启动时强等所有 peer 就绪。

### 13.2 拓扑版本号

**坑**：version 用 `uint64`，但如果不小心用了 `int`，溢出会出问题。**严格用 `uint64` 单调递增**。

**坑**：版本号比较只在**同一个 publisher_role** 范围内有意义。不要拿 A 的 version=5 和 B 的 version=3 比。

### 13.3 时钟问题

不同设备时钟可能不同步，所以**时间戳只能本地用**：
- 心跳判断超时：用本地"上次收到的时刻"
- 拓扑版本：用本地的 publisher 自己的单调计数器
- 不要跨设备比较时间戳

### 13.4 部分网络分区

A 和 B 都连着 C，但 A 和 B 之间断了。这时候 A 的本地拓扑只有 A→C 边，B 的只有 B→C 边。**Merge 后全局图是正确的**——A→C→B 可达。

### 13.5 共享内存的 ABA 问题

**坑**：读端读到一半，写端更新了。

**方案**：用 `version` 做版本号检查。读端读完之后检查 version 是否变化，如果变了重读一遍。

```cpp
RouteEntry RouteTableReader::Lookup(uint32_t dst) {
  while (true) {
    uint64_t v1 = shm_->version.load(std::memory_order_acquire);
    // 读 entries
    auto result = ...;
    uint64_t v2 = shm_->version.load(std::memory_order_acquire);
    if (v1 == v2 && (v1 % 2 == 0)) { // 偶数代表写完成
      return result;
    }
    // 重试
  }
}
```

写端用偶数表示"写完成"，奇数表示"正在写"。

### 13.6 单测覆盖率

Dijkstra 和拓扑融合很容易写出 bug，**强烈建议单测**：
- 单节点
- 两节点直连
- 三节点链状（A-B-C）
- 三节点环状（A-B-C-A）
- 四节点带备用路径
- 不连通图（A-B 和 C-D 两个孤岛）
- 动态加边
- 动态删边

## 十四、Demo 应用：分布式录音

为了让项目"显得真"，推荐做这个 demo：

- ✅ 文件传输（pub 大数据）

**展示价值**：拔掉一台 pi 的 USB，GUI 上自动只显示两路；插回去，自动恢复三路。
