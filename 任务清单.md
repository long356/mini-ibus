# mini-ibus 完整开发任务清单

*完整的开发任务清单，按 Phase 组织。每个任务设计为 30-60 分钟可完成，有明确输入/输出和测试要求。*

---

## 使用说明

### 任务格式

每个任务包含：
- **目标**：一句话说清楚要做什么
- **依赖**：依赖哪些已完成的任务
- **输入**：AI 需要读哪些文件
- **产出**：要写哪些文件
- **测试**：单元测试要覆盖什么
- **完成标准**：怎么判断任务完成

### 推荐工作流

1. 完成一个任务后，更新 CLAUDE.md 的"已完成模块"清单
2. 跑全量测试确保不破坏已有功能
3. git commit（建议每个任务一个 commit）
4. 进入下一个任务

### 任务编号规则

Phase号.任务序号，例如：
- 1.1 = Phase 1 的第 1 个任务
- 3.5 = Phase 3 的第 5 个任务

## 任务 1.14：RPC 超时 + 取消机制测试

**目标**：完善 Client::Call 的超时和取消语义，并写测试保证行为可控。

**依赖**：任务 1.10

**输入**：CLAUDE.md，任务 1.10 的 Client 实现

**产出**：

1. 确认 `Client::Call` 的接口语义：

```cpp
// 同步版本：阻塞直到收到响应或超时
ErrorCode Call(uint32_t target_addr, uint32_t service_id, uint32_t method_id,
               const std::string& args, std::string* result,
               std::chrono::milliseconds timeout = 5s);

// 异步版本（新增）
std::future<tl::expected<std::string, ErrorCode>>
CallAsync(uint32_t target_addr, uint32_t service_id, uint32_t method_id,
```

2. 完善内部实现：
   - 用 `std::promise/future` + sequence 号匹配
   - 超时后从 pending_requests 表中清理对应条目（避免内存泄漏）
   - Client 析构时所有 pending 请求都收到 Cancelled

3. `tests/unit/client_timeout_test.cc`：
   - **正常返回**：handler 100ms 返回，timeout=1s → 收到结果
   - **handler 慢**：handler 2s 返回，timeout=500ms → 收到 Timeout

   - **Client 析构时有 pending**：等待者收到 Cancelled
   - **并发多个 Call**：sequence 不冲突，每个收到正确结果

4. `tests/integration/test_rpc_timeout.sh`：

- 异步 future 等待行为正确

# Phase 5：业务侧增强 - MsgDispatcher 4 Lane（共 4 个任务，可选）

> **说明**：MsgDispatcher 不是 mini-ibus 框架的核心组件，而是**业务侧的最佳实践封装**。
> 它源自 IBus 的 GUI 进程设计，用于隔离慢任务和快任务，避免相互阻塞。
> 简单业务可以不使用，直接在 Client 的接收线程上处理消息。
>
> **本 Phase 把 MsgDispatcher 重新定位到 `addons/` 目录**，明确它不属于 core。
> 任务 1.6 中已实现的 MsgDispatcher 需要在本 Phase 移动并集成。

## 任务 5.1：MsgDispatcher 移到 addons/ + 集成进 Client

**目标**：把 MsgDispatcher 从 core 移到 addons，明确它是业务侧增强；并让 Client 可选地使用它。

**依赖**：Phase 4 完成、任务 1.6

**产出**：

1. **移动文件**：
   - 创建 `addons/` 目录
   - `core/msg_dispatcher.h` → `addons/msg_dispatcher.h`
   - `core/msg_dispatcher.cc` → `addons/msg_dispatcher.cc`
   - `tests/unit/msg_dispatcher_test.cc` 路径调整
   - 更新所有 include 路径和 CMakeLists.txt
   - `addons/README.md`：说明 addons 是业务侧可选增强

2. **Client 接口扩展**（在 core 中）：

```cpp
class Client {
public:
  // 设置自定义 dispatcher（可选）
  // 不设置时，所有 handler 在 Client 接收线程上同步执行
  using DispatcherFunc = std::function<void(std::function<void()>)>;
  void SetDispatcher(DispatcherFunc fn);

  // 为特定 topic 设置 dispatcher（覆盖全局）
  void SetTopicDispatcher(uint32_t topic_id, DispatcherFunc fn);
};
```

**注意**：Client 只暴露 `std::function<void(std::function<void()>)>` 这种通用接口，
**不直接依赖 MsgDispatcher**。这样 core 和 addons 完全解耦。

3. **业务侧使用方式**：

```cpp
// 简单业务：不用 dispatcher
Client client(config);
client.OnTopic(TOPIC_X, handler); // handler 在接收线程跑

// 复杂业务：用 MsgDispatcher
MsgDispatcher& dispatcher = MsgDispatcher::GetInstance();
dispatcher.Initialize();
client.SetTopicDispatcher(URGENT_TOPIC, [&](auto fn) {
  dispatcher.PostUrgent(fn);
});
client.SetTopicDispatcher(SLOW_TOPIC, [&](auto fn) {
  dispatcher.PostBlocking(fn);
});
```

4. `tests/unit/client_dispatcher_test.cc`：
   - 验证不设置 dispatcher 时，handler 在接收线程跑
   - 验证设置 dispatcher 后，handler 跑在 dispatcher 线程上
   - 验证不同 topic 可以用不同 dispatcher

**完成标准**：测试全过，core 不再 include addons
